# 开发路线图

> 目标：8～12 周内，从"C++ 基础 + STL"成长为可独立交付 C++ 大模型后端服务的工程师。

---

## 总体节奏

```
第 1-2 周  工程骨架
第 3-4 周  业务迁移
第 5 周    LLM 接入
第 6 周    性能与内存
第 7 周    并发改造
第 8 周    工程化收尾
第 9-12 周 RAG / 本地推理演进（可选）
```

---

## 第 1 周：CMake 工程搭建

**学习目标**：理解 CMake 项目结构，能够独立创建多模块工程。

**任务清单**：

- [ ] 阅读 `CMakeLists.txt`，理解 `add_library` / `add_executable` / `target_link_libraries`
- [ ] 编译本模板：`cmake -S . -B build && cmake --build build`
- [ ] 运行 `./build/CppRebuildTemplate`，观察 Stub 输出
- [ ] 运行测试：`cd build && ctest --output-on-failure`
- [ ] 尝试开启 `-DENABLE_SANITIZERS=ON` 重新编译

**新手常见坑**：

- CMake 路径区分大小写（Linux/macOS）
- `target_include_directories` 必须用 `PUBLIC`/`PRIVATE` 区分传递性

**验收标准（DoD）**：

- [ ] 本地编译零错误零警告
- [ ] 5 个单元测试全部通过

---

## 第 2 周：现代 C++ 特性强化

**学习目标**：熟练运用 C++11/14/17 的核心特性。

**任务清单**：

- [ ] 用 `std::unique_ptr` / `std::shared_ptr` 替换模板中所有裸指针
- [ ] 用 `std::optional<std::string>` 改造可能为空的返回值
- [ ] 用结构化绑定（`auto [key, val] = ...`）改写遍历
- [ ] 用 `std::string_view` 优化只读字符串参数
- [ ] 理解 Lambda 捕获规则，改写路由注册代码

**参考资料**：

- [cppreference.com](https://cppreference.com)
- [C++ Core Guidelines](https://isocpp.github.io/CppCoreGuidelines/)

**验收标准（DoD）**：

- [ ] 代码无裸 `new`/`delete`
- [ ] ASan 运行无报错

---

## 第 3 周：HTTP 服务框架接入

**学习目标**：用真实 HTTP 框架替换 Stub，实现可通过 curl 访问的接口。

**推荐框架**（任选其一，见 `docs/技术选型.md`）：

- [Crow](https://github.com/CrowCpp/Crow)（header-only，最易上手）
- [oat++](https://oatpp.io/)（中等复杂度）
- [Drogon](https://github.com/drogonframework/drogon)（功能最全）

**任务清单**：

- [ ] 选定框架，在 `CMakeLists.txt` 中用 FetchContent 引入
- [ ] 实现 `CrowHttpServer`（继承 `server::HttpServer`）
- [ ] 注册 `GET /health`、`POST /chat`、`POST /review` 路由
- [ ] 用 curl 验证：`curl localhost:8080/health`

**验收标准（DoD）**：

- [ ] 三个接口均可通过 curl 访问并返回 JSON
- [ ] 框架切换不影响 `core_lib` 和 `llm_lib` 编译

---

## 第 4 周：JSON 解析与业务迁移

**学习目标**：用 `nlohmann/json` 处理真实请求/响应 JSON，迁移核心业务逻辑。

**任务清单**：

- [ ] 引入 `nlohmann/json`（FetchContent）
- [ ] 用 JSON 解析 `/chat` 请求体的 `question` 字段
- [ ] 迁移作业批改规则（至少 3 条规则）：
  - 检测 `main` 函数是否存在
  - 检测是否有内存泄漏迹象（裸 `new` 而无 `delete`）
  - 代码行数统计
- [ ] 完善 `/review` 响应的 `score` 和 `comment` 字段

**验收标准（DoD）**：

- [ ] `/chat` 可正确提取并返回 `question`
- [ ] `/review` 可对 3 种代码模式给出不同评分

---

## 第 5 周：LLM 云 API 接入

**学习目标**：接入真实大模型 API，实现完整问答链路。

**任务清单**：

- [ ] 引入 `cpr`（HTTP 客户端库）
- [ ] 实现 `OpenAIClient::chat()` 的真实 HTTP 调用
- [ ] 通过环境变量读取 API Key：`std::getenv("OPENAI_API_KEY")`
- [ ] 实现超时控制（30 秒）与重试（3 次，指数退避）
- [ ] 引入 `spdlog`，添加结构化请求日志

**验收标准（DoD）**：

- [ ] `POST /chat {"question":"什么是递归？"}` 返回真实 LLM 回答
- [ ] API 调用日志可按级别过滤（info / error）

---

## 第 6 周：性能与内存优化

**学习目标**：系统识别性能热点，应用现代 C++ 优化手段。

**任务清单**：

- [ ] 对 `/chat` 添加请求耗时打点
- [ ] 实现 LRU 缓存（`std::list` + `std::unordered_map`）
- [ ] 审查字符串拼接，改用 `fmt::format` 或 `std::ostringstream`
- [ ] 运行 Valgrind / Heaptrack 检查内存分配热点
- [ ] 对比有/无缓存的 QPS（相同问题）

**验收标准（DoD）**：

- [ ] 缓存命中时 `/chat` 响应延迟 < 5 ms
- [ ] 无内存泄漏（Valgrind 零错误）

---

## 第 7 周：并发改造

**学习目标**：引入线程池，提升服务并发处理能力。

**任务清单**：

- [ ] 实现 `ThreadPool`（`std::thread` + `std::mutex` + `std::condition_variable`）
- [ ] HTTP 请求处理路由到线程池
- [ ] 用 `std::shared_mutex` 保护 LRU 缓存
- [ ] 压测（wrk/ab，并发 50/100），记录 QPS 和延迟
- [ ] 用 TSan 检查数据竞争

**验收标准（DoD）**：

- [ ] 并发 50 时无崩溃、无数据竞争
- [ ] QPS ≥ 100（不含 LLM 调用耗时）

---
