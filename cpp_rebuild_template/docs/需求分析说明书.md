## 1. 项目背景与问题定义

### 1.1 项目背景

原项目 `putianya/Python__GraduationProject` 是一个基于 Python + Vue 实现的"计算思维课程助手"，集成了大模型（LLM）问答、作业批改和知识检索功能。

本复刻项目目标：**用 C++ 重写后端服务层**，在保持相同业务功能的同时，系统性地学习现代 C++ 进阶能力。

### 1.2 目标用户

| 用户类型 | 描述 |
|---------|------|
| 主要用户（学习者） | 熟悉 C++ 基础语法与 STL，希望通过项目驱动进阶 |
| 次要用户（学生端） | 使用课程助手提问、提交作业的学生 |
| 教学管理者 | 查看作业批改结果、课程数据的教师 |

### 1.3 使用场景

- 学生在线提问：`POST /chat`
- 学生提交作业：`POST /review`
- 服务健康监控：`GET /health`
- 知识检索增强（RAG 演进阶段）：`POST /search`

### 1.4 成功标准（Definition of Done）

- [ ] 服务可通过 CMake 构建并运行
- [ ] 三个核心接口（`/health`、`/chat`、`/review`）返回正确响应
- [ ] 单元测试通过率 100%
- [ ] 接入真实大模型 API 后可完整处理一次问答

## 2. 功能需求

### 2.1 MVP 功能（最小可行产品）

#### F-01：健康检查

- **接口**：`GET /health`
- **描述**：返回服务当前状态，供运维和负载均衡探针使用
- **响应**：`{"status":"ok"}` 或 `{"status":"degraded"}`

- **接口**：`POST /chat`
- **请求体**：`{"question":"什么是递归？", "session_id":"可选"}`
- **描述**：将学生问题转发给 LLM，返回教学友好的回答
- **响应**：`{"answer":"...", "tokens_used":128}`

- **接口**：`POST /review`

- **描述**：对学生提交的代码/文本进行规则校验或 LLM 评审
- **响应**：`{"status":"ok", "score":85, "comment":"..."}`

## 3. 非功能需求

### 3.1 性能（Performance）

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| `/health` 响应延迟 | < 5 ms（P99） | 压测工具 wrk/ab |
| `/chat` 端到端延迟 | < LLM API 延迟 + 50 ms | 服务日志计时 |
| 并发吞吐量 | ≥ 100 QPS（不含 LLM 调用耗时） | 线程池压测 |

### 3.2 可靠性（Reliability）

- LLM API 调用失败时自动重试 3 次，指数退避
- 单个请求最大超时 30 秒
- 优雅关闭：接收 SIGTERM 后完成已有请求再退出

### 3.3 安全性（Security）

- 输入长度限制：`question` ≤ 4096 字符
- 不在日志中记录完整请求体（防止 API Key 泄露）
- 生产环境通过环境变量传入 API Key，禁止硬编码

### 3.4 可维护性（Maintainability）

- 编译零警告（`-Wall -Wextra -Wpedantic`）
- 代码覆盖率 ≥ 80%（核心模块）
- 每个模块职责单一，接口通过纯虚类（抽象接口）定义

### 3.5 可扩展性（Scalability）

- LLM 客户端通过接口隔离，支持一键切换模型厂商
- HTTP 框架通过抽象层隔离，支持替换底层框架
- 配置通过 JSON/YAML 文件管理，无需重新编译

### 3.6 可观测性（Observability）

- 结构化日志（spdlog JSON sink）
- 请求 ID（Request-ID）贯穿整个调用链
- 关键路径耗时打点

## 4. 架构设计

### 4.1 分层架构

```
┌────────────────────────────────────────────────┐
│              客户端（前端 / curl）               │
└──────────────────────┬─────────────────────────┘
                       │ HTTP/JSON
┌──────────────────────▼─────────────────────────┐
│              服务层（Server Layer）              │
│  StubHttpServer / Drogon / oat++ / Crow         │
│  路由分发、请求解析、响应序列化                    │
└──────────────────────┬─────────────────────────┘
                       │
┌──────────────────────▼─────────────────────────┐
│              核心业务层（Core Layer）             │
│  CourseAssistant：问答、评审、会话管理             │
└────────────┬─────────────────────┬─────────────┘
             │                     │
┌────────────▼──────┐   ┌──────────▼─────────────┐
│  LLM 接入层        │   │  数据层（Data Layer）    │
│  LLMClient 接口    │   │  配置、课程数据、会话存储  │
│  OpenAIClient      │   │  （内存 / SQLite / Redis）│
└───────────────────┘   └────────────────────────┘
```

### 4.2 模块职责

| 模块 | 路径 | 职责 |
|------|------|------|
| `server` | `src/server/` | HTTP 路由、请求/响应处理 |
| `core` | `src/core/` | 业务逻辑：问答、评审 |
| `llm` | `src/llm/` | LLM API 调用封装 |
| `include` | `include/` | 模块间公共接口（纯虚类） |

### 4.3 数据流（以 `/chat` 为例）

```
客户端
  → POST /chat {"question":"xxx"}
    → HttpServer 路由匹配
      → CourseAssistant::chat(question)
        → LLMClient::chat(ChatRequest)
          → HTTP POST → OpenAI API
          ← JSON 响应解析
        ← ChatResponse
      ← std::string answer
    ← HTTP 200 {"answer":"xxx"}
  ← 客户端展示
```

## 5. 并发与线程安全

### 5.1 线程池 / 任务队列思路

推荐使用生产者-消费者模型：

```cpp
// 伪代码示意
class ThreadPool {
    std::vector<std::thread> workers_;
    std::queue<std::function<void()>> tasks_;
    std::mutex mutex_;
    std::condition_variable cv_;
    bool stop_ = false;
public:
    void enqueue(std::function<void()> task);
    ~ThreadPool(); // 优雅关闭
};
```

HTTP 服务层每个请求封装成 `task` 投入线程池，避免阻塞主线程。

### 5.2 共享资源保护策略

| 共享资源 | 保护策略 |
|---------|---------|
| 会话缓存（SessionMap） | `std::shared_mutex`（读多写少） |
| LRU 缓存（回答缓存） | `std::mutex` + 双向链表 |
| 配置对象 | `const` 引用，初始化后只读 |
| 日志系统（spdlog） | 本身线程安全，无需额外锁 |

### 5.3 锁粒度建议

- **细粒度锁**：对每个独立资源加单独的锁，减少竞争
- **读写分离**：用 `std::shared_mutex` 允许并发读
- **避免死锁**：总是按相同顺序获取多把锁；使用 `std::lock()` 同时锁定多个 mutex

## 6. 内存与性能优化

### 6.1 RAII 与智能指针策略

| 场景 | 推荐 |
|------|------|
| 独占资源（HTTP 服务器实例） | `std::unique_ptr` |
| 共享资源（LLM 客户端） | `std::shared_ptr` |
| 避免循环引用 | `std::weak_ptr` |
| 裸指针 | 禁用（`new`/`delete` 不出现在业务代码） |

### 6.2 减少不必要拷贝

```cpp
// ❌ 按值传递大字符串
void process(std::string text);

// ✅ 按 const 引用
void process(const std::string& text);

// ✅ C++17 string_view（只读场景，零拷贝）
void process(std::string_view text);

// ✅ 移动语义（转交所有权）
void process(std::string&& text);
```

### 6.3 对象生命周期

- 优先使用栈分配（局部变量）
- `std::vector` 预分配 `reserve(n)` 避免多次扩容
- 避免在热路径上频繁分配/释放小对象（考虑对象池）

### 6.4 缓存策略

- LRU 缓存：对高频相同问题的 LLM 回答进行缓存
- 配置热更新：用 `std::atomic<bool>` 标记配置是否需要重新加载

### 6.5 基准测试建议

```bash
# 使用 Google Benchmark（可选依赖）
# 或简单自计时：
auto t0 = std::chrono::high_resolution_clock::now();
// ... 被测代码 ...
auto t1 = std::chrono::high_resolution_clock::now();
auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(t1-t0).count();
```

- [ ] 字符串/容器使用规范文档

### 新手常见坑

- **坑 1**：在循环中用 `+` 拼接 `std::string`——改用 `std::ostringstream` 或 `fmt::format`
- **坑 2**：`shared_ptr` 计数开销——独占场景用 `unique_ptr`

## 7. 大模型接入方案

### 7.1 路线 A：云 API 接入（推荐首选）

**流程**：

```
C++ 程序
  → libcurl / cpr（HTTP POST）
    → LLM API（OpenAI / DeepSeek / 通义千问）
      → JSON 响应
    ← nlohmann/json 解析
  ← ChatResponse
```

**关键实现要点**：

```cpp
// 请求体示例（伪代码）
nlohmann::json body = {
    {"model", "gpt-4o-mini"},
    {"messages", {
        {{"role","system"},{"content","你是计算思维课程助手"}},
        {{"role","user"},  {"content", userQuestion}}
    }},
    {"temperature", 0.7},
    {"max_tokens",  512}
};

// 发送 HTTP POST（使用 cpr）
auto resp = cpr::Post(
    cpr::Url{apiBaseUrl + "/v1/chat/completions"},
    cpr::Header{{"Authorization","Bearer " + apiKey},
                {"Content-Type","application/json"}},
    cpr::Body{body.dump()}
);
```

### 7.2 路线 B：本地推理（进阶）

使用 [llama.cpp](https://github.com/ggerganov/llama.cpp)：

1. 下载量化模型（GGUF 格式）
2. 调用 `llama.cpp` 的 C API 或启动其内置 HTTP 服务
3. 复用 `LLMClient` 接口，实现 `LlamaCppClient`

```
演进路径：
OpenAIClient (云 API)
    ↓ 接口不变，替换实现
LlamaCppClient (本地推理)
```

### 7.3 路线 C：RAG 渐进式演进

```
阶段 1：纯 LLM 问答（本模板当前阶段）
阶段 2：加入文档检索（关键词 / BM25）
阶段 3：向量化（embedding API 或 fastembed）
阶段 4：向量数据库（FAISS C++ 接口 / Qdrant HTTP API）
阶段 5：检索 + 重排 + 生成（完整 RAG 管线）
```

## 8. 构建调试规范

### 8.1 Debug / Release 区分

```bash
# Debug（含调试信息，开 Sanitizer）
cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug -DENABLE_SANITIZERS=ON
cmake --build build/debug

# Release（优化 O2，去除断言）
cmake -S . -B build/release -DCMAKE_BUILD_TYPE=Release
cmake --build build/release
```

### 8.2 编译警告配置

```cmake
# CMakeLists.txt 中已配置：
add_compile_options(-Wall -Wextra -Wpedantic)
# 目标：编译零警告（将警告视为错误可加 -Werror）
```

### 8.3 Sanitizer

| Sanitizer | 检测内容 | 启用方式 |
|-----------|---------|---------|
| ASan（AddressSanitizer） | 堆越界、use-after-free | `-DENABLE_SANITIZERS=ON` |
| UBSan（UndefinedBehaviorSanitizer） | 未定义行为（整数溢出等） | 同上 |
| TSan（ThreadSanitizer） | 数据竞争 | 单独配置（不与 ASan 同用） |

### 8.4 静态分析建议

```bash
# clang-tidy
clang-tidy src/**/*.cpp -- -std=c++17 -Iinclude

# cppcheck
cppcheck --enable=all --std=c++17 src/
```
